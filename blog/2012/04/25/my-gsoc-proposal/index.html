My GSoC Proposal

Apr 25th, 2012

As mentionned before, I’ve been accepted to do a Google Summer of Code for GNOME. My proposal is about the creation of a new library providing models and widgets to display and choose contacts. For those who don’t want to read the complete proposal here are the interesting bits :

What is the ultimate goal of your proposal?

The ultimate goal of my proposal is to develop a new C library providing models and Gtk+ widgets to display and choose contacts that could be used by any GNOME application, a feature that Empathy’s developers are considering to implement since a long time. It has been discussed last year during the IM/Social Hackfest and some notes are available on the wiki.

The contact chooser would be integrable in a lot of GNOME application or plugins such as Nautilus-sendto, the Shell’s contact search engine, Evolution (which could begin to use Folks and allow to send mails to IM contacts), or any GNOME application which could benefit from IM integration using Telepathy tubes for example. The library will be usable in C, Python and Javascript thanks to the GObject Introspection support.

What components/modules will the proposed work modify or create?

A model of the loaded contacts will be developed and widgets will be built on top of it. This will be developed as a new library that other projects will be able to use. This library will probably be named Folks-gtk, but this name might change during the development if we find a cooler one. The API will mainly provide :
-Functionalities to fetch and display all contacts from Folks
-An API to define if one or more contacts can be picked
-An API to let the user filter out contacts
-Helper code for some common filters (like online contacts)

This will, obviously, be related to Empathy, Telepathy and Folks. To validate the library and its API, Empathy’s nautilus-sendto plugin will be modified to make use of it. If time permits, I’ll also modify Shell’s contact search feature to use it as well.

What benefits does your proposed work have for GNOME and its community?

  • IM integration and Telepathy tubes :
    This contact chooser I’m proposing to implement here would help improving IM’s integration in the whole GNOME desktop experience. Whatever the application is and whatever the developer wants to do involving IM contacts, he won’t have to think about the way to choose contacts or struggle implementing it. Of course, it’s not only valuable for developers but it’s also profitable for the end users since more applications could start offering more social features.
    Even if Telepathy tubes have been implemented several years ago they are only quite rarely used in GNOME. Vinagre/Vino use them for Empathy’s desktop sharing, a really popular and innovating feature, but that’s about it. There are at least two reasons which could explain this : the first is that tubes are quite difficult to use, and the second is the difficulty to fetch and present the list of contacts (needed in almost any application implementing social features). Telepathy’s team is working to fix the first problem (telepathy-glib now has high level tubes API) and my project would fix the second one.

  • Contacts coherence :
    Since GNOME 3.2, gnome-contacts defined a new way to display contacts in GNOME. Empathy’s developers are working with the design team to re-design Empathy’s contact list to make it more coherent with the rest of the desktop (see bug #669484). If more applications are going to integrate social features it’s important to stay coherent and offer a coherent way to display contacts in the whole GNOME desktop.
